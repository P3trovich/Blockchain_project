# Обоснование архитектуры PharmaTrack

Документ объясняет, почему система собрана именно из этих кубиков, и что
с ней может пойти не так.

---

## 1. Общая схема

Система состоит из трёх слоёв:

```
┌─────────────────────────────────────────────────────────┐
│  Клиенты:  QR-сканер (пациент)   Web UI (участники)     │
│            мобильное приложение  IoT-симулятор           │
└────────────┬───────────────────────┬────────────────────┘
             │                       │
             │ ethers.js / WalletConnect / web3.py
             ▼                       ▼
┌─────────────────────────────────────────────────────────┐
│  Блокчейн-слой:  смарт-контракт PharmaTrack.sol          │
│                  EVM-сеть (Ethereum / Polygon / consortium)│
└────────────┬────────────────────────────────────────────┘
             │ events (DrugRegistered, TemperatureRecorded…)
             ▼
┌─────────────────────────────────────────────────────────┐
│  Off-chain:  FastAPI-шлюз (опц.), индексер событий,      │
│              IoT-симулятор, кэш для тяжёлых выборок      │
└─────────────────────────────────────────────────────────┘
```

Клиенты ходят либо напрямую в контракт (read-only вызовы — `verifyDrug`,
`getDrug`), либо через off-chain шлюз для тяжёлых выборок (история партии,
агрегации). Запись всегда идёт напрямую в контракт от подписи участника.

## 2. Почему смарт-контракт — единственный источник истины

В исходной репе ledger жил в Python и хранил состояние в `dict`. Это
удобно для прототипа, но на защите такой ledger — это просто база данных,
обёрнутая в слово «блокчейн». Поэтому всё состояние мы перенесли в
Solidity-контракт. Off-chain слой не хранит ничего важного: он умеет
кэшировать и индексировать, но если его удалить — система продолжит
работать, читая напрямую из цепочки.

Это даёт два важных свойства:

1. **Невозможность безмолвной правки.** Любое изменение состояния — это
   подписанная транзакция в публичном (или хотя бы консорциумном)
   реестре. Откатить изменение задним числом нельзя — можно только
   написать сверху новую транзакцию, которая тоже останется в истории.
2. **Внешняя проверяемость.** Аптека, регулятор, пациент могут запустить
   свою ноду и убедиться в состоянии без посредников. У централизованной
   базы такого свойства нет в принципе.

## 3. Почему именно EVM, а не Hyperledger

Выбор между EVM и Hyperledger Fabric — главная развилка в проекте.
Аргументы за Fabric: permissioned по дизайну, частные каналы между парами
участников, привычные для корпораций инструменты. Аргументы против:
закрытость от внешних проверяющих (пациент с QR-сканером не подключится к
Fabric-сети просто так), узкая экосистема, высокий порог входа.

EVM проигрывает по «корпоративности», но выигрывает по совместимости и
скорости разработки. Конкретные следствия для проекта:

- Любой пользователь с MetaMask может проверить препарат за 0 рублей и
  без регистрации в системе.
- Используется индустриальный стек (Solidity, Hardhat, ethers.js,
  OpenZeppelin), которым легко найти разработчиков.
- Если в будущем потребуется приватность транзакций между парами участников
  (например, цена партии), можно перейти на zk-rollup (Polygon zkEVM,
  Scroll) или прикрутить commit-reveal схему — без переписывания всего.

## 4. Состояние в контракте

Контракт хранит:

- `mapping(string => Drug) drugs` — все упаковки по серийному номеру.
  Структура `Drug` содержит идентификаторы, температурный диапазон,
  владельца, стадию (`Manufacturer / Distributor / Pharmacy / Sold`),
  флаг блокировки и счётчик нарушений.
- `mapping(string => TempRecord[]) tempLog` — лог температурных замеров.
- `mapping(string => TxRecord[]) txLog` — лог всех событий по препарату
  (тип, отправитель, момент времени, хэш доп.данных). Сами данные в
  событие не пишутся — для экономии storage достаточно их хэша, а само
  содержимое восстанавливается из логов событий.
- `mapping(bytes32 => mapping(address => bool)) hasRole` — простая ACL.

Сознательный компромисс: мы не используем OpenZeppelin AccessControl, чтобы
не тащить внешнюю зависимость в учебный проект. Логика ACL занимает 5 строк
и проверяется глазами.

Ещё компромисс: температуры хранятся как `int16` в десятых долях градуса
(`50` = 5.0°C). Это бережёт газ и storage, и для фарм-применений точности
0.1°C достаточно — стандартные требования к холодовой цепи указываются как
раз с такой точностью.

## 5. Контроль доступа

Пять ролей: MANUFACTURER, LOGISTICS, DISTRIBUTOR, PHARMACY, REGULATOR.
Роли выдаёт админ контракта (в прототипе — деплоер, в продакшене —
multisig регулятора и крупных участников). Каждая публичная функция,
меняющая состояние, помечена модификатором `only(ROLE)`.

`verifyDrug` — публичный метод, доступ не нужен. Любой клиент может
прочитать статус препарата без транзакции и без газа.

Слабое место: связка «адрес ↔ юрлицо» живёт off-chain, у админа.
Криптографически это никак не заверено. Решение для пилота — хранить
рядом таблицу «адрес → выписка из ЕГРЮЛ» с подписью администратора и
ссылкой на off-chain документ. В продакшене — DID/SSI поверх системы,
но это уже следующая итерация.

## 6. Off-chain слой

В исходной репе FastAPI был основным контейнером логики. Сейчас его роль:

- **API-шлюз** для тех клиентов, которым неудобно ходить в EVM напрямую
  (например, мобильное приложение, которое предпочитает обычный REST).
  Шлюз вызывает контракт через web3.py и кэширует ответы.
- **Индексер событий.** Слушает Solidity-events и складывает их в
  PostgreSQL для быстрых выборок («все нарушения температуры в Москве за
  май»). Самая частая причина такого слоя — тяжёлые отчёты, которые из
  чистого контракта читать дорого.
- **IoT-симулятор** в `src/scripts/iot_simulator.py` — генерирует
  показания температуры с заданным распределением и подписывает их
  ключом «логистического оператора». Полезен для демо и для нагрузочных
  тестов.

Если off-chain слой упадёт, проверка препарата в аптеке всё равно
работает (через прямой вызов контракта).

## 7. Безопасность

Ключевые угрозы и реакция:

- **Подделка подписи.** Невозможна на уровне криптографии — каждый
  вызов подписан приватным ключом, проверяется самим контрактом через
  `msg.sender`.
- **Регистрация дубликата серийника.** Контракт проверяет уникальность
  и реверит транзакцию. Покрыто тестом.
- **Передача в аптеку при сломанной холодовой цепи.** Контракт
  блокирует переход, если `breachCount > 0`. Покрыто тестом.
- **Незаконная блокировка препарата конкурентом.** Роль REGULATOR
  выдаётся только проверенному адресу. В продакшене — multisig 3-из-5.
- **Захват админ-аккаунта.** Самый опасный сценарий: атакующий начинает
  раздавать роли. Контр-меры — multisig для admin (нет в прототипе,
  обязателен в продакшене), периодическая ротация ключей, мониторинг
  событий `RoleGranted`.
- **Reentrancy и прочая EVM-классика.** В нашем контракте нет переводов
  средств, нет внешних вызовов, поэтому reentrancy неприменим. Все
  публичные функции линейны и не вызывают чужие контракты.

## 8. Что мы НЕ закрыли в этой версии

Чтобы было честно:

- Нет on-chain верификации лицензий MANUFACTURER (полагаемся на off-chain
  выдачу роли).
- Нет защиты от спама от честного, но скомпрометированного логиста (он
  может писать тысячи замеров температуры). Решение — лимиты или
  плата за газ от его же кошелька; в прототипе не реализовано.
- Структура `Drug` упирается в stack-too-deep при компиляции, поэтому
  включён `viaIR`. Это нормально, но в проде стоит разнести структуру
  на несколько маппингов — будет дешевле по газу.
- Не реализовано право на забвение для самого факта «препарат куплен
  таким-то адресом». Сейчас этого нет в данных вообще — просто потому,
  что мы не пишем покупателя. Если в будущем потребуется учёт
  льготных рецептов — это придётся продумывать отдельно с юристами.
